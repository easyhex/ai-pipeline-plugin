# Как работать в Claude Code — Полный гайд по пайплайну

> На основе статьи Игоря Масленникова (3000+ часов в Claude Code)

---

## Философия: почему именно так

Claude Code генерирует код быстро, но без дисциплины — это красивые галлюцинации. Модель может уверенно написать код по устаревшему API, заявить "готово" без проверки, пропустить тесты.

Три плагина решают три проблемы:

| Плагин | Роль | Вопрос |
|---|---|---|
| **Superpowers** | Методология | **КАК** работать |
| **Beads** | Память проекта | **ЧТО** делать |
| **Template Bridge** | Каталог специалистов | **КТО** поможет |

Дополнительные MCP-серверы:

| MCP | Роль |
|---|---|
| **Context7** | Актуальная документация библиотек (не из обучающей выборки) |
| **Sequential Thinking** | Пошаговое рассуждение для сложных задач |
| **Playwright** | Глаза — открыть браузер, проверить результат визуально |

## Пайплайн: от задачи до готового кода

### Шаг 1. Создай эпик в Beads

Эпик — это контейнер для намерения, не подробная задача.

```
/beads:create -t epic "Добавить JWT авторизацию"
```

или в терминале:

```bash
bd create -t epic "Добавить JWT авторизацию"
```

**Зачем:** Beads хранит задачи в git. Они переживают перезапуск сессии, компактификацию контекста, переключение между терминалами. Без этого — через 2 часа ты заново объясняешь Claude, что делаешь.

---

### Шаг 2. Brainstorming — думай до кода

```
/superpowers:brainstorming
```

Claude **не бросается писать код**, а:
- Задаёт уточняющие вопросы
- Предлагает 2-3 подхода
- Рисует дизайн
- Ждёт твоего одобрения

**Пример:** Ты говоришь "добавь авторизацию". Claude спрашивает: JWT или сессии? Access + refresh или только access? Хранить в куках или localStorage? Какой TTL? Нужен ли rate limiting?

**Зачем:** Экономия часов отладки. Без brainstorming Claude пишет "по ощущениям" и через час ты понимаешь, что нужно было совсем другое.

---

### Шаг 3. Plan — декомпозиция на мелкие задачи

```
/superpowers:writing-plans
```

После brainstorming Claude разбивает работу на задачи по **2-5 минут** каждая. Не "сделать авторизацию", а:

1. Создать таблицу tokens в БД
2. Написать middleware верификации
3. Добавить refresh token логику
4. Написать endpoint /login
5. Написать endpoint /refresh

---

### Шаг 4. Создай подзадачи и зависимости в Beads

```bash
bd create "Схема таблицы tokens"
bd create "Middleware верификации"
bd create "Refresh token логика"
bd dep add middleware tokens        # middleware зависит от таблицы
bd dep add refresh tokens           # refresh зависит от таблицы
```

**Граф зависимостей:** Beads автоматически строит граф. `bd ready` покажет "фронт готовности" — задачи, у которых все зависимости выполнены. Закрыл одну — автоматически открылись зависимые.

---

### Шаг 5. Изоляция (для нетривиальной работы)

```
/superpowers:using-git-worktrees
```

Создаёт изолированную git worktree. Ты работаешь в отдельной ветке, основной код не трогается.

---

### Шаг 6. Реализация — TDD цикл

Берём задачу из фронта готовности:

```bash
bd ready                           # что можно делать прямо сейчас?
bd update --claim TASK-ID          # беру эту задачу
```

Затем — **обязательный** цикл Red-Green-Refactor для каждой задачи:

#### RED — напиши падающий тест

```
/superpowers:test-driven-development
```

Claude пишет тест на желаемое поведение. Запускает. Тест **должен упасть**. Если не падает — тест бесполезен.

#### GREEN — минимальный код

Claude пишет **минимальный** код, чтобы тест прошёл. Не идеальный, не красивый — просто рабочий.

#### REFACTOR — чистка

Код чистится, тесты остаются зелёными.

#### COMMIT — фиксация

Результат коммитится. Задача в Beads закрывается:

```bash
bd close TASK-ID --reason "Таблица tokens создана, миграция работает"
```

**Почему TDD обязателен:** Claude генерирует быстро, но без тестов ты не можешь отличить работающий код от галлюцинации. Тесты — единственный объективный арбитр.

---

### Шаг 7. Code Review

```
/superpowers:requesting-code-review
```

Claude проверяет свой же код на баги, проблемы безопасности, производительность.

---

### Шаг 8. Верификация — доказательства до утверждений

```
/superpowers:verification-before-completion
```

**Принцип:** Claude не имеет права заявить "готово", пока не:

1. Определит команду, доказывающую результат
2. Запустит её (свежую, не из кэша)
3. Прочитает полный вывод
4. Убедится, что вывод подтверждает заявление

Не "я думаю, что работает", а "вот вывод команды, вот доказательство".

---

### Шаг 9. Завершение ветки

```
/superpowers:finishing-a-development-branch
```

Claude предлагает варианты: merge, PR, или cleanup.

---

### Шаг 10. Закрытие эпика

```bash
bd close EPIC-ID --reason "JWT авторизация реализована"
```

## Дополнительные инструменты — когда и как

### Context7 — актуальная документация

**Когда использовать:** При работе с любой библиотекой или фреймворком.

**Проблема без него:** Claude уверенно пишет код по документации полугодовой давности. API изменился, параметры переименовались — он не знает.

**Как:**

```
/context7:docs next.js app router
```

Или автоматически — в `CLAUDE.md` прописано правило "всегда проверяй Context7 перед реализацией с библиотекой".

**Must have для:** Next.js, Supabase, и любых активно развивающихся библиотек.

---

### Sequential Thinking — сложные рассуждения

**Когда использовать:** Архитектурные решения, сложная отладка, когда нужно держать в голове несколько гипотез.

**Как работает:** Claude раскладывает задачу пошагово, может пересмотреть предыдущий шаг, разветвить рассуждение. Вместо "прыжка к выводу" — структурированный анализ.

Вызывается автоматически Claude при необходимости через MCP.

---

### Playwright — проверка в браузере

**Когда использовать:** Фронтенд-разработка.

**Что умеет:**
- Открыть страницу
- Сделать скриншот
- Кликнуть по элементу
- Заполнить форму
- Прочитать DOM

**Зачем:** Не "я думаю, что CSS правильный", а "я открыл страницу и проверил". Claude перестаёт быть слепым.

---

### Template Catalog — специалисты по запросу

**Когда использовать:** Когда задача выходит за рамки текущих скиллов.

```
/template-bridge:template-catalog
```

413+ специализированных агентов в 26 категориях: security audit, data science, blockchain, game dev и т.д.

Claude проактивно предложит: "Для аудита безопасности есть security-auditor — установить?"

Агент ставится одной командой, используется для задачи, после — можно удалить. Всегда свежая версия из репо.

## Что происходит автоматически (хуки)

### При старте сессии (SessionStart)

1. `bd prime` — Beads подтягивает состояние проекта: открытые задачи, блокировки, историю
2. Напоминание о workflow

**Результат:** Новая сессия за секунды получает полный контекст. Не нужно заново объяснять, что делаешь.

### Перед компактификацией (PreCompact)

1. `bd prime` — сохраняет состояние задач перед тем, как Claude "забудет" старые сообщения

**Результат:** Задачи никогда не теряются, даже при длинных сессиях.

## Быстрая шпаргалка

```
Задача пришла
    ↓
bd create -t epic "Цель"              ← Beads: зафиксируй намерение
    ↓
/superpowers:brainstorming             ← Думай: что, зачем, как
    ↓
/superpowers:writing-plans             ← Декомпозиция на 2-5 мин задачи
    ↓
bd create + bd dep add                 ← Beads: подзадачи + зависимости
    ↓
/superpowers:using-git-worktrees       ← Изоляция (если нетривиально)
    ↓
bd ready → bd update --claim           ← Beads: взял задачу
    ↓
/superpowers:test-driven-development   ← RED: падающий тест
    ↓
Реализация                             ← GREEN: минимальный код
    ↓
Рефакторинг                            ← REFACTOR: чистка
    ↓
Коммит + bd close                      ← Фиксация результата
    ↓
(повторить для каждой подзадачи)
    ↓
/superpowers:requesting-code-review    ← Проверка
    ↓
/superpowers:verification-before-completion ← Доказательства
    ↓
/superpowers:finishing-a-development-branch ← Merge/PR
    ↓
bd close EPIC-ID                       ← Готово
```

## Главное правило

> **AI — мультипликатор. Мультипликатор нуля — ноль. Мультипликатор десятки — сотня.**

Чем сильнее ты как инженер — тем больше получаешь от Claude Code. AI не заменяет проектирование, декомпозицию, архитектурные решения. AI заменяет набивание символов.

Superpowers с обязательным TDD — не блажь, а необходимость. Без тестов невозможно отличить работающий код от красивой галлюцинации.
