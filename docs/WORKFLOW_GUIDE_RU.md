# Полный гайд по пайплайну ai-pipeline

> Один плагин для Claude Code, 7 пользовательских команд, автоматический критик на двух стадиях, 5 слоёв контекста.

---

## Философия: почему так, а не иначе

Claude Code генерирует код быстро, но без дисциплины — это красивые галлюцинации. Без тестов нельзя отличить рабочий код от сгенерированного «по ощущениям». Без явного дизайна Claude через час пишет совсем не то. Без памяти о проекте каждая сессия начинается с нуля.

Этот плагин решает все три проблемы одним пайплайном:

| Слой | Что даёт |
|---|---|
| **Дизайн перед кодом** | brainstorming → spec → plan, до того как написана первая строчка |
| **TDD как закон** | ни одной строки production-кода без падающего теста сначала |
| **Критик на двух стадиях** | senior-инженер автоматически ревьюит и спеку, и дифф |
| **5 слоёв памяти** | Beads (задачи) + Lessons (баги) + Master Plan (что строим) + Context7 (внешние либы) + Serena memory (конвенции и решения) |
| **Авто-захват уроков** | каждый `/fix` пишет урок, чтобы тот же баг никогда не стоил дважды |

## Установка (один раз на машине)

```bash
# 1. Сам плагин
claude plugin marketplace add easyhex/ai-pipeline-plugin
claude plugin install ai-pipeline@ai-pipeline-marketplace

# 2. CLI инструменты для функциональности (если ещё не стоят)
brew install beads        # для постоянного трекинга задач
brew install uv           # для Serena (Python менеджер)

# 3. Serena (память проекта) — /init поставит автоматически с твоего согласия,
#    но можно и заранее:
uv tool install -p 3.13 serena-agent@latest --prerelease=allow
serena init
claude mcp add --scope user serena -- serena start-mcp-server --context claude-code --project-from-cwd
```

После этого 7 команд (`/init`, `/plan-improve`, `/feature`, `/improve`, `/fix`, `/lesson`, `/remember`) и subagent `senior-critic` доступны глобально в любой сессии Claude Code.

## Создание нового проекта

Одной командой:

```bash
mkdir my-app && cd my-app
claude
```

Внутри Claude Code:

```
> /init "todo-приложение с realtime синхронизацией"
```

Что происходит автоматически:

1. **Pre-flight** — проверяет, что папка пустая (без существующего `package.json`, `src/`, `CLAUDE.md`)
2. **Пререкизиты** — детектит отсутствующие плагины (superpowers, beads, context7-plugin, template-bridge) + Serena, спрашивает одно `[y/n]`, ставит всё что не хватает
3. **3 уточняющих вопроса** — стек / целевой пользователь / приоритет качества
4. **Запись файлов** — заполняет `docs/architecture.md`, `docs/features.md`, `docs/roadmap.md`, кладёт `CLAUDE.md`, `docs-meta/PIPELINE.md`, `docs-meta/LESSON_FORMAT.md`, `.gitignore`, `.claude/settings.json`
5. **`bd init`** — инициализирует трекер задач
6. **`git init` + первый коммит** — фиксирует scaffold
7. **Отчёт** — что записано, что дальше

Никакого языкоспецифичного скаффолдинга — `/init` НЕ создаёт `package.json`, `pyproject.toml` и т.п. Это ты сам делаешь привычным способом (`create-next-app`, `uv init`, `cargo new` — что любишь). Пайплайн агностичен к стеку.

## Пайплайн фичи: одна команда, 11 внутренних фаз

```
> /feature "добавить регистрацию пользователей через magic link"
```

И ты можешь идти пить кофе. Внутри происходит:

| Фаза | Что делает |
|---|---|
| 1. **Ground** | Читает `docs/{architecture,features,roadmap}.md`, все `.claude/lessons/*.md`, релевантные `.serena/memories/*.md`. Дёргает Context7 за актуальной докой библиотек, которые есть в `package.json`. |
| 2. **Brainstorm** | Через `superpowers:brainstorming` — вопросы (если есть), 2-3 подхода, спека → `docs/superpowers/specs/<дата-slug>.md` |
| 3. **Critic gate-1** | `senior-critic` ревьюит спеку: пропущенные edge cases, конфликт с master plan, нарушения уроков. Отчёт → `docs/superpowers/critic-reports/<slug>-gate1.md` |
| 4. **Plan** | Через `superpowers:writing-plans` — декомпозиция на задачи 2-5 минут |
| 5. **Beads tasks** | `bd create` для каждой подзадачи + граф зависимостей через `bd dep add` |
| 6. **Worktree** (если >3 задач) | `git worktree add` — изоляция в отдельной ветке |
| 7. **TDD loop** | Для каждой задачи: RED → verify-fail → GREEN → verify-pass → REFACTOR → `git commit` |
| 8. **Critic gate-2** | `senior-critic` ревьюит дифф: пропущенные тесты, дыры безопасности, нарушения уроков. Опционально предлагает что записать в Serena memory. |
| 9. **Verify** | Запускает proving commands (тесты, lint, typecheck, build) — fresh, не из кэша. Читает exit code. |
| 10. **Finish** | По `pipeline.finish_mode`: или merge в main + удаление worktree, или `gh pr create` с критик-отчётом в body |
| 11. **Master Plan update** | Двигает фичу в `docs/features.md` секции "Shipped" с датой; закрывает beads-эпик |

Ты вмешиваешься **только** в двух случаях:
- brainstorm задаёт реальный уточняющий вопрос (не «как назовём переменную», а «авторизация — JWT или session cookies?»)
- критик нашёл Critical findings — выбираешь `continue` / `address` / `override`

Всё остальное — автоматически.

## 7 пользовательских команд

| Команда | Когда |
|---|---|
| `/init "<описание>"` | Один раз — bootstrap проекта |
| `/plan-improve "<изменение>"` | Уточнить мастер-план (без написания кода) |
| `/feature "<описание>"` | **Новая** функциональность — полный авто-пайплайн |
| `/improve "<изменение>"` | **Изменение поведения** существующей фичи — тот же авто-пайплайн |
| `/fix "<баг>"` | Отладка через `superpowers:systematic-debugging` + автозапись урока |
| `/lesson` | Ручная запись урока (редко — обычно `/fix` пишет сам) |
| `/remember "<факт>"` | Записать факт о проекте в Serena memory (обычно критик предлагает сам) |

Ты **не** вводишь `/brainstorm`, `/plan`, `/build`, `/critic`, `/verify`, `/finish` — это внутренние фазы команд выше.

### `/feature` vs `/improve` vs `/fix` — где граница

- **`/feature`** — добавляешь функциональность, которой не было (новый endpoint, новая страница, новая интеграция)
- **`/improve`** — меняешь поведение существующей фичи (например: «сделай /login запоминающим устройство»). Не добавляет новую запись в `features.md` — обновляет существующую.
- **`/fix`** — баг. Что-то работало, перестало работать или работает неправильно. Заканчивается записью урока с полями `trigger`/`symptom`/`root_cause`/`prevention`.

Если по ходу `/improve` оказалось, что это на самом деле новая фича — пайплайн остановится и попросит тебя запустить `/feature`.

## 5 слоёв контекста

Каждый слой решает свою задачу — **не пересекаются**:

| Слой | Где живёт | Что хранит |
|---|---|---|
| **Beads** | `.beads/` | Задачи + граф зависимостей. `bd ready` всегда показывает что можно делать сейчас. |
| **Lessons** | `.claude/lessons/*.md` | Правила-предотвращения багов. Каждый урок: trigger / symptom / root_cause / prevention + 2-3 предложения контекста. Жёсткий лимит — 3 предложения. |
| **Master Plan** | `docs/{architecture,features,roadmap}.md` | Что строим: целевая архитектура (≤300 строк), список фич (≤500), roadmap (≤200) |
| **Context7** | живой MCP-запрос | Актуальная документация внешних библиотек (Next.js, Prisma, и т.д.) |
| **Serena memory** | `.serena/memories/*.md` | Стабильные знания о проекте: конвенции (`snake_case в БД, camelCase в TS`), дизайн-решения с rationale (`Postgres вместо Mongo потому что X`), доменные факты (`биллинг в EUR`) |

**Правило выбора слоя:**
- Баг? → урок
- Задача? → beads
- Архитектурное решение? → master plan (`architecture.md`)
- Внешняя библиотека? → Context7 (или Serena memory если это PROJECT-специфичное использование)
- Конвенция или ambient knowledge? → Serena memory

## Критик: твой автоматический senior

`senior-critic` — Claude subagent, который запускается автоматически на двух стадиях:

**Gate-1 (после спеки)** ищет:
- Невысказанные предположения
- Пропущенные edge cases (auth fail, пустые input, concurrent writes, network errors)
- Scope creep сверх того, что просил пользователь
- Конфликты с master plan (нарушения архитектуры, конфликты с уже отгруженными фичами)
- Нарушения уроков
- Дыры в плане тестов

**Gate-2 (после диффа)** ищет:
- Поведение, заявленное в спеке, но не реализованное в коде или тестах
- Проблемы безопасности (auth bypass, валидация input, секреты, injection)
- Дыры в обработке ошибок
- Тесты, которые проходят, но ничего не проверяют
- Дрейф архитектуры

Также критик добавляет секцию `## Memories to capture (suggested)` — что стоит записать в Serena memory. Пайплайн авто-пишет эти предложения.

Тон — спокойный senior, не адверсариальный devil's advocate. Никаких «возможно», «вероятно» — либо нашёл проблему, либо нет.

Критик НЕ одобряет и не отклоняет работу. Он выдаёт findings — ты решаешь:
- `continue` — принять, идти дальше
- `address` — обратно на предыдущую фазу с findings как доп-инпут (макс 2 цикла)
- `override` — принять риск, оставить причину в отчёте

Если Critical findings и ты молчишь — по умолчанию `address`.

## Уроки: каждый баг учит навсегда

`/fix "<баг>"` всегда заканчивается записью урока. Формат (`docs-meta/LESSON_FORMAT.md`):

```markdown
---
trigger: писание prisma миграций на таблицах с существующими данными
symptom: миграция прошла локально, упала на проде с NOT NULL violation
root_cause: prisma сгенерил ALTER TABLE с NOT NULL без DEFAULT; локально таблица пустая
prevention: Всегда проверяй сгенерированный SQL на NOT NULL колонки без DEFAULT. Если есть — исправь миграцию: 3 шага (nullable, backfill, NOT NULL).
---

Добавили колонку `last_seen_at`. Prisma сгенерил ALTER TABLE с NOT NULL без DEFAULT. Локально окей (таблица пустая), на проде — миллион строк, миграция упала. Фикс — 3-шаговая миграция.
```

Уроки автоматически загружаются в каждой ground-фазе. Критик их цитирует на каждой стадии. Через несколько недель пайплайн перестаёт повторять старые ошибки.

## Память Serena: что отличает от уроков и master plan

Serena memory — для **стабильного знания о проекте**, которое не подходит ни под Beads (это задача), ни под Lessons (это не баг), ни под Master Plan (это не архитектура), ни под Context7 (это не внешняя либа).

Примеры подходящих фактов:
- «Используем `snake_case` в колонках БД, `camelCase` в TypeScript»
- «Auth модуль обходит `/healthz` — middleware пропускает этот path»
- «Биллим юзеров в EUR, не USD»
- «Pagination всегда в контроллере, никогда в модели»

Что НЕ записываем в Serena:
- Баги → это уроки
- Задачи → это beads
- Внешняя дока (Next.js App Router) → это Context7
- «Какая у нас архитектура» → это `docs/architecture.md`

**Кто пишет:**
- Критик автоматически на gate-2 (`/feature`, `/improve`) и gate-1 (`/plan-improve`) — предлагает, пайплайн пишет
- Ты вручную через `/remember "<факт>"`

**Кто читает:** каждая ground-фаза `/feature` и `/improve` подгружает релевантные memories.

## Хуки

В `~/.claude/settings.json` рекомендованы:

```json
{
  "hooks": {
    "SessionStart": [
      { "matcher": "", "hooks": [
        { "type": "command", "command": "bd prime 2>/dev/null || true" },
        { "type": "command", "command": "echo \"Lessons: $(ls .claude/lessons/*.md 2>/dev/null | wc -l | tr -d ' ') | Pipeline: /init /plan-improve /feature /improve /fix /lesson /remember\"" }
      ]}
    ],
    "PreCompact": [
      { "matcher": "", "hooks": [
        { "type": "command", "command": "bd prime 2>/dev/null || true" }
      ]}
    ]
  }
}
```

`/init` пишет это в `.claude/settings.json` нового проекта автоматически. Пример для глобальной установки лежит в `settings.example.json`.

**Что они делают:**
- **SessionStart** — `bd prime` подгружает состояние Beads (открытые задачи, блокировки) + echo напоминает о пайплайне и количестве уроков
- **PreCompact** — `bd prime` сохраняет состояние перед компактификацией контекста, чтобы задачи не потерялись

## Шпаргалка

```
Новый проект:
  mkdir app && cd app && claude
  > /init "<описание>"
       │
       ↓ автоматически: install plugins → 3 вопроса → файлы → bd init → git init

Новая фича:
  > /feature "<описание>"
       │
       ↓ автоматически: ground → brainstorm → critic → plan → beads → TDD → critic → verify → finish

Изменение существующей фичи:
  > /improve "<изменение>"
       │
       ↓ тот же пайплайн, но НЕ добавляет новую фичу в features.md

Баг:
  > /fix "<баг>"
       │
       ↓ автоматически: systematic-debugging → critic → fix → verify → write lesson

Конкретный факт о проекте:
  > /remember "<факт>"
       │
       ↓ записывает в .serena/memories/

Уточнить мастер-план без кода:
  > /plan-improve "<изменение>"
```

## Жёсткие правила (per-project `CLAUDE.md`)

После `/init` файл `CLAUDE.md` в проекте требует от Claude:

1. Никакого production-кода без падающего теста сначала
2. Никаких «готово» без запуска проверочной команды и чтения её вывода
3. Перед любой фазой — прочитать `.claude/lessons/` и применить уроки, у которых `trigger:` подходит к текущей задаче
4. Перед использованием любой библиотеки — Context7 запрос
5. Critic gate-1 (после спеки) и gate-2 (после диффа) — автоматически
6. Каждый зелёный TDD-цикл заканчивается `git commit`; каждый `/fix` — записью урока
7. Перед `/feature` или `/improve` — прочитать `.serena/memories/` и применить релевантные

## Git: что автоматизировано, что ручное

| Действие | Кто делает |
|---|---|
| `git init`, `git add`, `git commit` | пайплайн |
| Создание ветки (`feature/<slug>`, `improve/<slug>`) | пайплайн |
| `git worktree add` (для нетривиальных задач) | пайплайн |
| Локальный merge ветки в main | пайплайн |
| `gh pr create` (если `pipeline.finish_mode = "pr"`) | пайплайн |
| `git push` к remote | **ты, вручную** |

`git push` остаётся за тобой — это публикация, ты должен решать когда отправлять.

## Главное правило

> **AI — мультипликатор. Мультипликатор нуля — ноль. Мультипликатор десятки — сотня.**

Чем сильнее ты как инженер — тем больше получаешь от Claude Code. Этот плагин не заменяет проектирование, декомпозицию, архитектурные решения. Он заменяет ручное управление пайплайном (когда какой агент дёрнуть, какой файл прочитать, какой коммит сделать) — и принуждает к дисциплине (TDD, верификация, критик), без которой быстрый AI-код превращается в красивые галлюцинации.

7 команд. Внутри — 11 фаз и 5 слоёв памяти. Снаружи — `/feature "..."` и кофе.
